## Completed Work (summary)

### B3 -- Restructuring Improvements

Three additions to `go-cfg-to-structured` in `goast/prim_restructure.go`:
labeled break for switch/select returns in loops, multiple return value synthesis,
limited forward-goto elimination. See `memory/2026-03-24-transformation-primitives-design.md`.

### C2 -- Dataflow Analysis Framework

Worklist-based forward/backward dataflow analysis over SSA block graphs.
Implemented in `(wile goast dataflow)`.

**Actual API** (differs from original object-oriented design):

```scheme
(run-analysis direction lattice transfer ssa-fn [initial-state] ['check-monotone])
;; direction:  'forward or 'backward
;; lattice:    from (wile algebra lattice)
;; transfer:   (block state) -> state (per-block, not per-instruction)
;; ssa-fn:     SSA function from go-ssa-build
;; Returns:    result alist ((idx in out) ...)

(analysis-in result block-idx)   ;; query in-state
(analysis-out result block-idx)  ;; query out-state
(analysis-states result)         ;; full result alist
```

Key decisions: SSA blocks only (no separate CFG block provider), per-block transfer
(not per-instruction), eager result alist (not lazy object), reverse postorder traversal.
`defuse-reachable?` left as-is (structurally different: flat iteration, not block-aware).

See `memory/2026-03-26-c2-dataflow-design.md`, `memory/2026-03-26-c2-dataflow-impl.md`.

### C3 -- Pre-built Abstract Domains

Five domains in `(wile goast domains)`: reaching definitions, liveness, constant
propagation, sign analysis, interval analysis. Each provides a lattice + factory
function that calls `run-analysis` and returns the result alist.

`go-concrete-eval` evaluates Go SSA integer opcodes on Scheme integers (reused by C5).

See `memory/2026-03-26-c3-domains-design.md`, `memory/2026-03-26-c3-domains-impl.md`.

### C4 Semiring Path Algebra

`(wile goast path-algebra)`: semiring-parameterized Bellman-Ford over call graphs.
Lazy single-source computation with per-source caching. Exports: `make-path-analysis`,
`path-analysis?`, `path-query`, `path-query-all`.

See `memory/2026-04-12-path-algebra-design.md`, `memory/2026-04-12-path-algebra-impl.md`.

## C4 CFL-Reachability -- Open

Context-sensitive path analysis via matched call/return pairs. Separate from
semiring path algebra (which is context-insensitive).

### Grammar

```
path    := matched path | path path
matched := call_i path ret_i | epsilon
```

### Implementation

Summary edges (Reps 1998). Worklist propagates summaries when matched pairs
are found. Each call site `i` has paired `call_i` / `ret_i` labels.

### API

```scheme
(make-cfl-reachability call-graph)
(cfl-reachable? analysis "A" "D")   ;; -> #t if context-sensitive path exists
(cfl-paths analysis "A" "D")        ;; -> list of matched call chains
```

### Dependencies

- `go-callgraph` (existing)
- Call sites need unique identifiers -- extract from SSA call instructions

### Use case

Function boundary redrawing. `cfl-paths` reveals which boundaries are
semantic (multiple calling contexts) vs accidental (single context,
candidate for inlining). Complements `(wile goast fca-recommend)` which
detects boundary problems from field access patterns.

### Library placement

New library `(wile goast cfl)` or extend `(wile goast path-algebra)`.
CFL-reachability isn't a semiring computation (it's a pushdown analysis),
so a separate library is cleaner despite the conceptual kinship.

## C5 -- Galois Connections and Auto-Lifting -- Open

Automatic generation and validation of abstract transfer functions.

### Foundation: `(wile algebra galois)`

Already available in wile. Exports:

```scheme
make-galois-connection  ;; construct (alpha, gamma) pair with lattices
galois-connection?      ;; type predicate
gc-alpha                ;; abstraction function
gc-gamma                ;; concretization function
gc-concrete-po          ;; concrete partial order
gc-abstract-lattice     ;; abstract lattice
gc-sound?               ;; validate soundness: alpha(f(x)) <= f#(alpha(x))
```

### Abstract domain protocol

```scheme
(make-abstract-domain
  name              ;; 'sign, 'interval, 'flat, 'powerset
  lattice           ;; the abstract lattice
  galois-connection ;; from (wile algebra galois)
  gamma-finite?     ;; can gamma enumerate all representatives?
  gamma-sample      ;; (abstract-value -> list of concrete values)
  abstract-eval     ;; #f (auto-generate) or hand-written
  concrete-eval)    ;; go-concrete-eval from C3 (in (wile goast domains))
```

### Auto-lifting

```scheme
(auto-lift domain opcode)  ;; -> (abstract-args -> abstract-value)
```

Strategy selection:
- `gamma-finite?` true + `abstract-eval` is `#f` -> exhaustive: for each
  pair of abstract inputs, compute all gamma combinations, apply concrete-eval,
  join alpha results
- `abstract-eval` provided -> use it, validate via sampling

**Go-specific advantage:** Fixed-width integers. For int8, gamma(+) = {1,...,127} --
finite. Exhaustive verification against int8, trust by monotonicity for wider types.

### Soundness validation

```scheme
(validate-transfer domain opcode n-samples)
```

For random concrete inputs `(x, y)`, verify:
`alpha(f(x, y)) <= f#(alpha(x), alpha(y))`

Catches bugs in hand-written interval transfer functions. Confirms exhaustive
sign tables against wider types.

**Retroactive C3 validation:** After C5 is built, run `validate-transfer` against
all C3 hand-written transfer tables (sign arithmetic, interval arithmetic).

### Domain registry

```scheme
(register-abstract-domain! domain)
(lookup-abstract-domain name)
(list-abstract-domains)
```

Used by C6 to look up domains when promoting beliefs.

### Library placement

`(wile goast abstract-domain)`. Depends on `(wile algebra galois)`,
`(wile goast domains)` (for `go-concrete-eval`), `(wile goast dataflow)`.

### Scope

`make-abstract-domain` owns the domain: lattice + Galois connection + transfer tables.
It does not own the worklist or traversal — `run-analysis` (C2) handles that.

```
domain = lattice + abstraction + transfer tables
analysis = domain + direction + SSA function   (composed at call site via run-analysis)
```

`auto-lift` generates transfer tables from the domain; `run-analysis` consumes them.

## C6 -- Belief Graduation -- Open

### V1: Beliefs -> assertions

Beliefs with 100% adherence become dataflow assertions via C2's `run-analysis`.

```scheme
(graduate-belief belief-name package-pattern)
;; 1. Run belief; if adherence < 100%, return #f
;; 2. Compile checker into run-analysis call with appropriate lattice
;; 3. Return result alist for per-path analysis
```

Checker-to-assertion mapping:

| Checker | Lattice | Direction |
|---------|---------|-----------|
| `paired-with "A" "B"` | 2-element: `{called-A, paired}` | forward |
| `checked-before-use "val"` | Boolean | forward |
| `ordered "A" "B"` | 2-element: `{seen-A, seen-B}` | forward |
| `co-mutated "f1" "f2"` | Powerset over `{f1, f2}` | forward |

**Granularity upgrade:** Normal beliefs check per-function. Graduated assertions
check per-path -- finer granularity, stronger guarantee.

### Integration with `run-beliefs`

```scheme
(run-beliefs package-pattern #:graduate #t)
;; normal results + graduated dataflow analysis for 100% beliefs
```

The graduated analysis calls `run-analysis` directly (C2's actual API), not through
any intermediate object protocol. Each checker maps to a specific lattice and
transfer function, then `run-analysis` runs it per-function.

### Dependency on C5 (optional)

C6 v1 works without C5 -- the lattice and transfer for each checker type are
hand-written (they're simple 2-4 element lattices). C5's domain registry becomes
useful only if users define custom graduated beliefs (v2 -- deferred).

### Future: Belief-defined lattices (v2 -- documented, not built)

The belief DSL becomes a front-end for defining custom abstract domains:

```scheme
(define-belief "resource-lifecycle"
  (sites (functions-matching (contains-call "Open")))
  (expect (lifecycle "Open" "Use" "Close"))
  (threshold 1.0 3)
  (graduate-as
    (lattice (flat-lattice '(unopened opened used closed) eq?))
    (transitions
      ("Open"  'unopened -> 'opened)
      ("Use"   'opened  -> 'used)
      ("Close" 'used    -> 'closed))))
```

**Deferred because:** Requires transition DSL design, integration with
`make-abstract-domain`, and validation that user-defined lattices are actually
lattices (monotonicity check). Graduation-only (v1) delivers real value without
this complexity.

### Library placement

`(wile goast belief-graduate)` or extend `(wile goast belief)`.
Extending belief.scm is simpler (it already has access to belief internals),
but adds a dependency on `(wile goast dataflow)` -- which belief.sld already
imports. So extending is feasible and avoids a new file.
