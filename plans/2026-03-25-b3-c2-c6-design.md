# B3 + C2-C6 Design

Restructuring improvements (B3) and algebra-based static analysis framework (C2-C6).

## Sequencing

B3 implementation runs first (parallel with C2 design). Then C2→C3→C4→C5→C6 linearly.

```
Phase 1: B3 impl + C2 design
Phase 2: C2 impl
Phase 3: C3 impl
Phase 4: C4 impl
Phase 5: C5 impl
Phase 6: C6 impl
```

Each piece gets its own implementation plan.

## B3 — Restructuring Improvements

Three additions to `go-cfg-to-structured` in `goast/prim_restructure.go`.

### B3.1: Labeled break (switch/select returns in loops)

**Current behavior:** `replaceReturnsInStmts` encounters a `SwitchStmt`/`TypeSwitchStmt`/
`SelectStmt` containing a return → bails with `false`.

**Change:** Instead of bailing:
1. Assign a label to the enclosing loop: `_loop<N>:` (new counter, separate from `_ctl<N>`)
2. Replace returns inside switch/select cases with `_ctl<N> = K; break _loop<N>`
   (labeled break targets the loop, not the switch)
3. Guard-ifs after the loop work exactly as they do now

**AST construction:** New helpers `makeLabeledFor(label, forStmt)`,
`makeLabeledBreak(label)`. The label wraps the existing for/range in `ast.LabeledStmt`.

**Edge case:** Nested switches inside loops — the labeled break always targets the
innermost enclosing loop, which is already the one being rewritten. No ambiguity.

### B3.2: Multiple return values

**Problem:** When a return references loop-local variables:
```go
for _, v := range items {
    if v < 0 { return v, errNeg }  // v is loop-scoped
}
```

After rewriting, `v` isn't in scope for the guard-if.

**Solution:**
1. Detect when return expressions reference loop-local variables
2. Synthesize result variables: `var _r0 <type>; var _r1 <type>` before the loop
3. Replace `return v, errNeg` with `_r0 = v; _r1 = errNeg; _ctl0 = 1; break`
4. Guard-if becomes `if _ctl0 == 1 { return _r0, _r1 }`

**Type inference:** Use function's `FuncType.Results` from the AST — return types are
in the declaration. No dependency on type-checking.

### B3.3: Goto / labeled branches

**Current behavior:** `containsGoto` detects any `token.GOTO` or `ast.LabeledStmt` → returns `#f`.

**Approach:** Limited goto elimination (not the full Erosa & Hendren algorithm):

1. **Forward-only gotos** (goto jumps to a label later in the function) → rewrite as
   `if !_jumped { ... } else { <label-body> }`. Handles the common "goto cleanup" pattern.
2. **Backward gotos** (loops) → wrap in `for { ... if !cond { break } }`. Less common in Go.
3. **Cross-branch gotos** (goto into/out of if/switch branches) → still return `#f`.
   Genuinely hard and rare in Go.

**Key constraint:** Labels used only by `goto` (not by `break`/`continue`) are the
restructuring targets. Labels used by `break`/`continue` are already structured.

## C2 — Dataflow Analysis Framework

Extension of `(wile goast dataflow)` from narrow def-use reachability to a general
dataflow analysis framework.

### Block protocol

The framework operates over an abstract block protocol, with two implementations:

```scheme
(block-id block)           ;; → unique identifier
(block-succs block)        ;; → list of successor blocks
(block-preds block)        ;; → list of predecessor blocks
(block-instrs block)       ;; → list of instructions (SSA instrs or AST stmts)
(block-entry provider)     ;; → entry block
(block-exit provider)      ;; → exit block (or #f)
```

- **`ssa-block-provider`** — wraps SSA function blocks from `go-ssa-build`
- **`cfg-block-provider`** — wraps CFG blocks from `go-cfg`

### Analysis combinator

```scheme
(make-analysis
  direction        ;; 'forward or 'backward
  lattice          ;; from (wile algebra)
  transfer         ;; (instruction × state) → state
  block-provider)  ;; ssa-block-provider or cfg-block-provider
```

Returns an analysis object:
- `(run-analysis analysis)` → map from block-id to (in-state, out-state)
- `(query-analysis analysis block-id)` → state at that block

### Traversal

- **Forward:** reverse postorder from entry
- **Backward:** postorder from exit
- Block ordering computed once via DFS on successor graph

### Worklist algorithm

```
initialize: all blocks get lattice bottom
add entry (forward) or exit (backward) to worklist
while worklist non-empty:
    pick block with best priority (block ordering)
    compute new out-state = transfer(in-state, block-instrs)
    if out-state changed (lattice-leq?):
        add successors (forward) or predecessors (backward)
```

### Map lattice

Per-variable analysis: `(map-lattice key-universe value-lattice)` lifts a value lattice
pointwise over a set of keys.

```scheme
(define state-lattice (map-lattice vars (flat-lattice constants equal?)))
```

### Monotonicity assertion

Debug mode flag on `make-analysis`. After each transfer, verifies `old ⊑ new`.
Raises error with block-id and instruction if violated.

### Relationship to existing exports

Current `dataflow.scm` exports stay unchanged. They become consumers:

| Current export | Becomes |
|----------------|---------|
| `boolean-lattice` | Utility |
| `ssa-all-instrs` | Used by `ssa-block-provider` |
| `ssa-instruction-names` | Used by powerset analyses |
| `make-reachability-transfer` | One specific transfer function |
| `defuse-reachable?` | Thin wrapper over `make-analysis` + `run-analysis` |

## C3 — Pre-built Abstract Domains

Four lattices that plug into C2 as the `lattice` parameter. Each provides the lattice
plus a `make-<name>-analysis` factory.

### Powerset

Already exists in `(wile algebra)`. C3 adds analysis-ready wrappers:

- **`make-reaching-definitions`** — forward, gen/kill per instruction
- **`make-liveness`** — backward, use/def per instruction

### Flat

Already exists as `(flat-lattice elements equal?)`. C3 adds:

- **`make-constant-propagation`** — forward, fold when all operands concrete, otherwise ⊤

Requires `go-concrete-eval` — SSA opcode evaluator on Scheme values. Defined once in
C3, reused by C5's auto-lifting.

```scheme
(go-concrete-eval opcode args)
;; opcode: 'add, 'sub, 'mul, 'div, 'rem, 'and, 'or, 'xor, ...
;; args: list of Scheme values
;; returns: result or 'unknown
```

### Sign

New 5-element lattice: `{⊥, -, 0, +, ⊤}`.

- **`make-sign-analysis`** — forward, sign arithmetic lookup table (25 entries per binop)
- Transfer table hand-written for C3; validated by C5 retroactively

### Interval

New lattice: `(lo . hi)` pairs plus ⊥ and ⊤.

- **`make-interval-analysis`** — forward, interval arithmetic, requires widening
- Widening strategy: if a bound grew, push to ±∞
- `widening-threshold` parameter controls iterations before widening

### Shared structure

All four: lattice + `make-<name>-analysis` factory + transfer function dispatching
on instruction tag (SSA: `ssa-binop`, `ssa-phi`, `ssa-call`, `ssa-if`; AST: `assign-stmt`,
`if-stmt`, `call-expr`).

## C4 — Path Algebra on Call Graphs

Semiring-based computation over call graphs.

### Generic path computation

```scheme
(make-path-analysis
  semiring          ;; from (wile algebra)
  call-graph        ;; from go-callgraph
  edge-weight)      ;; (edge → semiring-value), default: semiring-one
```

Computes semiring "sum over all paths" between nodes. Path value = `times` along
edges, multiple paths combined with `plus`. Algebraic path problem (Tarjan 1981).

### Boolean semiring — reachability

```scheme
(define reach (make-path-analysis (boolean-semiring) cg (lambda (_) #t)))
(path-query reach "main" "panic")  ;; → #t or #f
```

Subsumes `go-callgraph-reachable`.

### Tropical semiring — shortest/longest call chains

```scheme
(define depth (make-path-analysis (tropical-semiring) cg (lambda (_) 1)))
(path-query depth "Handler" "DB.Query")  ;; → 3
```

### CFL-reachability — context-sensitive analysis

Tracks matched call/return pairs. Grammar:

```
path    := matched path | path path
matched := call_i path ret_i | ε
```

Implementation via summary edges (Reps 1998). Worklist propagates summaries when
matched pairs are found.

```scheme
(make-cfl-reachability call-graph)
(cfl-reachable? analysis "A" "D")   ;; → #t if context-sensitive path exists
(cfl-paths analysis "A" "D")        ;; → list of matched call chains
```

**Use case:** Function boundary redrawing. `cfl-paths` reveals which boundaries are
semantic (multiple calling contexts) vs accidental (single context, candidate for inlining).

## C5 — Galois Connections and Auto-Lifting

Automatic generation and validation of abstract transfer functions using
`(wile algebra galois)`.

### Abstract domain protocol

```scheme
(make-abstract-domain
  name              ;; 'sign, 'interval, 'flat, 'powerset
  lattice           ;; the abstract lattice
  galois-connection ;; (α, γ) pair
  gamma-finite?     ;; can γ enumerate all representatives?
  gamma-sample      ;; (abstract-value → list of concrete values)
  abstract-eval     ;; #f (auto-generate) or hand-written
  concrete-eval)    ;; go-concrete-eval from C3
```

### Auto-lifting

```scheme
(auto-lift domain opcode)  ;; → (abstract-args → abstract-value)
```

Strategy selection:
- `gamma-finite?` true + `abstract-eval` is `#f` → exhaustive enumeration: for each
  pair of abstract inputs, compute all γ combinations, apply concrete-eval, join α results
- `abstract-eval` provided → use it, validate via sampling

**Go-specific advantage:** Fixed-width integers. For int8, γ(+) = {1,...,127} — finite.
Exhaustive verification against int8, trust by monotonicity for wider types.

### Soundness validation

```scheme
(validate-transfer domain opcode n-samples)
```

For random concrete inputs `(x, y)`, verify:
`α(f(x, y)) ⊑ f#(α(x), α(y))`

The fundamental soundness condition. Catches bugs in hand-written Interval transfer
functions. Confirms exhaustive Sign tables against wider types.

### Retroactive validation of C3

After C5 is built, run `validate-transfer` against all C3 hand-written transfer tables.
Closes the loop: C3 provides tables, C5 proves soundness.

### Domain registry

```scheme
(register-abstract-domain! domain)
(lookup-abstract-domain name)
(list-abstract-domains)
```

Used by C6 to look up domains when promoting beliefs.

## C6 — Belief Graduation

### V1: Beliefs → assertions

Beliefs with 100% adherence become dataflow assertions.

```scheme
(graduate-belief belief-name package-pattern)
;; 1. Run belief; if adherence < 100%, return #f
;; 2. Compile checker into C2 analysis with appropriate lattice
;; 3. Return analysis object for independent execution
```

Checker-to-assertion mapping:

| Checker | Lattice | Direction |
|---------|---------|-----------|
| `paired-with "A" "B"` | 2-element: `{called-A, paired}` | forward |
| `checked-before-use "val"` | Boolean | forward |
| `ordered "A" "B"` | 2-element: `{seen-A, seen-B}` | forward |
| `co-mutated "f1" "f2"` | Powerset over `{f1, f2}` | forward |

**Granularity upgrade:** Normal beliefs check per-function. Graduated assertions
check per-path — finer granularity, stronger guarantee.

### Integration

```scheme
(run-beliefs package-pattern #:graduate #t)
;; normal results + graduated dataflow analysis for 100% beliefs
```

### Future: Belief-defined lattices (v2 — documented, not built)

The belief DSL becomes a front-end for defining custom abstract domains:

```scheme
(define-belief "resource-lifecycle"
  (sites (functions-matching (contains-call "Open")))
  (expect (lifecycle "Open" "Use" "Close"))
  (threshold 1.0 3)
  (graduate-as
    (lattice (flat-lattice '(unopened opened used closed) eq?))
    (transitions
      ("Open"  'unopened → 'opened)
      ("Use"   'opened  → 'used)
      ("Close" 'used    → 'closed))))
```

**Deferred because:** Requires transition DSL design, integration with
`make-abstract-domain`, and validation that user-defined lattices are actually lattices.
Graduation-only (v1) delivers real value without this complexity.
