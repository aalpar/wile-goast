> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `cmd/wile-goast/scripts/wile-axis-b.scm` — a Scheme script that reads the 475-primitive manifest (`wile/plans/axis-b-manifest.scm`), narrows each primitive's result-writing sinks via `go-ssa-narrow`, classifies the results into one of seven buckets, and emits raw + markdown inventories. Invocation: `wile-goast --run wile-axis-b`.

**Architecture:** The real analysis work happens in `go-ssa-narrow` (PR-2). The Scheme script is deliberately thin glue: manifest parsing, SSA walking to find sink calls, per-primitive result merging, Go→wile type mapping, bucket classification, output formatting.

**Upstream artifacts:**

- `go-ssa-narrow` primitive (PR-2) — narrows an SSA value name to type set + confidence + reasons.
- `current-go-target` parameter (PR-1) — lets the script implicit-target wile's repo.

**Parent designs:**

- Wile repo: `plans/2026-04-19-axis-b-analyzer-design.md` §5 — the seven buckets, §7 — the output formats.

**Sink functions (finalized — §7.5 research):**

Wile's `CallContext` interface (`wile/machine/call_context.go`) exposes exactly two result-writing methods:

- `github.com/aalpar/wile/machine.CallContext.SetValue(v values.Value)`
- `github.com/aalpar/wile/machine.CallContext.SetValues(vs ...values.Value)`

Primitives that type-assert to `*MachineContext` for full VM access call the same methods on the concrete type:

- `(*github.com/aalpar/wile/machine.MachineContext).SetValue`
- `(*github.com/aalpar/wile/machine.MachineContext).SetValues`

Two logical sinks, four SSA-visible names (interface-invoke + concrete-call per sink). Well under the 15-name kill threshold (§9 of parent design).

## Task 2: Sink-call enumeration

**Files:**
- Modify: `cmd/wile-goast/scripts/wile-axis-b.scm` — add `find-sink-calls`, `sink-name?` predicates.

**Approach:** For each primitive's SSA function, walk blocks + instrs. A sink call is an `ssa-call` whose `func` field (static) or `method` field (invoke) matches one of the four sink names. Record the first operand (args[0]) — that's the value being written.

- [ ] **Step 1: Define the sink-names constant**

```scheme
(define sink-names
  ;; Fully-qualified function/method names that write to the result.
  ;; See plans/2026-04-19-pr-3-axis-b-script-impl.md sink research.
  '("(*github.com/aalpar/wile/machine.MachineContext).SetValue"
    "(*github.com/aalpar/wile/machine.MachineContext).SetValues"
    "(github.com/aalpar/wile/machine.CallContext).SetValue"
    "(github.com/aalpar/wile/machine.CallContext).SetValues"))
```

Also add a coarser membership check that matches by suffix (`"SetValue"`/`"SetValues"`) so slight formatting differences in SSA's `String()` output don't fall through. The strict list stays as documentation.

- [ ] **Step 2: `find-sink-calls` walks function SSA**

Input: an `(ssa-func ...)` alist. Output: list of `(sink-call (sink <name>) (value-arg <name-string>))` — the value-arg name is what `go-ssa-narrow` consumes.

Commit: "feat(axis-b): enumerate sink calls in SSA functions".

## Task 4: Per-primitive narrowing pipeline

**Files:**
- Modify: script — add `analyze-primitive`, result-merge helpers.

Given a resolved `(ssa-func ...)` alist:
1. Call `find-sink-calls` → list of (sink, value-arg-name).
2. For each, call `(go-ssa-narrow ssa-func value-arg-name)` → narrow-result alist.
3. Merge results across sink calls (union types, widened-dominates confidence, union reasons).

- [ ] **Step 1: `narrow-sink-call ssa-func sink-call`** returns one narrow-result.
- [ ] **Step 2: `merge-narrow-results rs`** mirrors Go-side `mergeResults` logic:
  - Union all `types` lists, deduplicate.
  - Confidence: `widened` if any input widened, else `no-paths` if all no-paths, else `narrow`.
  - Union all `reasons` lists, deduplicate.
- [ ] **Step 3: `analyze-primitive entry func-index`** orchestrates the above and returns a per-primitive record tagged `axis-b-entry`.

Commit: "feat(axis-b): per-primitive narrowing pipeline".

## Task 6: Bucket classification

**Files:**
- Modify: script — add `classify-bucket` per §5 of the parent design.

Seven buckets per the parent design. Decision tree:
1. If confidence `no-paths` → `Side-effecting`.
2. If confidence `widened` → `Helper-widened`.
3. If narrowed set has exactly 1 type → `Single`.
4. If narrowed set matches `{T, #f-or-nil}` shape → `Maybe(T)` — check for `Boolean` (specifically `*Boolean:#f`) or `EmptyList` paired with a single concrete T. Note: since `go-ssa-narrow` records type strings, the `#f` pattern shows up as `"*values.Boolean"`; rely on the `nil-constant` reason tag if present, or the `narrow` confidence with `{"*values.Boolean", T}` shape.
5. If narrowed set has 2-3 types → `Narrow-union`.
6. If narrowed set has 4+ types → `Broad-union`.
7. `Polymorphic` — reserved for primitives whose type depends on input (identity, `car`, `apply`). Signal: narrowed set contains `types.Value` (the interface) or only `parameter` reason. For this MVP, collapse into `Helper-widened` unless a cleaner signal emerges — we're measuring distribution, not making fine distinctions that don't have consumers yet.

- [ ] **Step 1: Write `classify-bucket narrowed-types confidence reasons`** returning a bucket symbol.
- [ ] **Step 2: Unit-test the classifier with a small in-Scheme table** (no SSA needed) — assert the decision tree covers each bucket.

Commit: "feat(axis-b): classify narrowed results into seven buckets".

## Kill criteria (restated from parent §9)

**First-run checkpoint** — if any of these trigger, stop and revisit:

- **>30% Helper-widened** (>128 of ~428 non-binding-only primitives): PR-2 narrowing is under-powered for wile's primitives. Analyze the reason-tag distribution, open a PR-2' to extend narrowing for the dominant tag. Do NOT land the inventory.
- **Any no-paths on non-binding-only primitives** (>5): a sink was missed in the §7.5 research. Re-read wile/machine for missed writers.
- **Sink-call count per primitive averaging <0.5** (i.e., most primitives show no sinks): either the sink names are formatted differently in SSA output than we expected, or there's a whole class of primitives that doesn't write results through the CallContext interface (unlikely but possible — e.g., primitives that return errors exclusively).

Each trigger converts PR-3 back into a tooling-design question before continuing.
